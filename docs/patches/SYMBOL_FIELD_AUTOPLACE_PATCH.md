# Задача: читаемое авторазмещение полей символа при повороте

## Проблема

В KiCad поля символа (`Reference`, `Value` и пользовательские поля) при первом размещении из библиотеки могут быть автоматически расставлены читаемо. Но после ручной правки и последующего поворота символа поля часто вращаются вместе с графикой и становятся вертикальными или визуально нечитабельными.

На практике это выглядит как поломка чертежного стиля: дезигнаторы и значения, повернутые набок вместе с символом, ухудшают читаемость схемы и создают ощущение, что программа "живет своей жизнью".

Текущая логика смешивает разные пользовательские намерения:

- пользователь красиво разместил поле для текущей ориентации символа;
- пользователь хочет, чтобы поле навсегда вращалось вместе с символом;
- пользователь действительно вручную задал нестандартную ориентацию поля.

Эти сценарии должны обрабатываться по-разному.

## Цель

Улучшить поведение полей символа при повороте и зеркалировании:

- поля с включенным авторазмещением должны оставаться читаемыми;
- `Reference` по умолчанию должен размещаться сверху от символа;
- `Value` по умолчанию должен размещаться снизу от символа;
- перемещение всего символа не должно отключать или переопределять авторазмещение полей;
- ручная раскладка поля должна рассматриваться как применимая к конкретной ориентации, а не как универсальная геометрия для всех углов.

## Желаемое поведение

### Авторазмещение

Если у поля включено авторазмещение:

- при первом размещении символа поле ставится по правилам авторазмещения;
- при повороте символа поле пересчитывается;
- при зеркалировании символа поле пересчитывается;
- текст остается читаемым;
- перемещение всего символа не меняет режим поля.

Базовый стиль:

- `Reference` располагается над bounding box символа;
- `Value` располагается под bounding box символа;
- оба поля остаются горизонтальными/readable, если нет явного пользовательского override.

### Ручная правка поля

Ручное перемещение поля не должно автоматически означать "вращать это поле вместе с символом при всех будущих поворотах".

Более корректная модель:

- пользователь двигал поле напрямую: сохранить позицию для текущей ориентации;
- пользователь повернул поле напрямую: сохранить ориентацию для текущей ориентации или перевести поле в более сильный locked/manual режим;
- пользователь двигал весь символ: не менять состояние полей;
- пользователь повернул весь символ: использовать override для новой ориентации, если он есть, иначе авторазмещение.

Пример:

```text
0 градусов:
  пользователь вручную поставил Reference сверху, Value снизу

поворот на 90 градусов:
  если ручной позиции для 90 градусов нет, применить авторазмещение

возврат на 0 градусов:
  восстановить ручную раскладку для 0 градусов
```

## Возможная модель состояний

Текущего деления на auto/manual может быть недостаточно. Желательная модель:

```text
Auto
Manual per orientation
Manual locked/global
```

### Auto

Поле управляется алгоритмом авторазмещения.

```text
rotate/mirror symbol -> recalculate field placement
```

### Manual per orientation

Пользовательская позиция валидна только для конкретной ориентации символа.

```text
0 degrees   -> user override A
90 degrees  -> user override B or auto fallback
180 degrees -> user override C or auto fallback
270 degrees -> user override D or auto fallback
```

### Manual locked/global

Поле явно заблокировано пользователем и должно сохранять ручное поведение при трансформациях. Этот режим нужен для редких случаев, когда пользователь действительно хочет нестандартную ориентацию/позицию поля.

## Ограничение по формату

Перед реализацией обязательно изучить текущий формат `.kicad_sch` и `.kicad_sym`.

Изменение формата допустимо только как upstream proposal для следующей версии KiCad и должно приниматься разработчиками проекта. Для патча текущей версии желательно не менять формат файлов и не ломать обратную совместимость.

## Этап 1: исследование текущей модели

### Формат `.kicad_sch`

Выяснить:

- как хранится позиция поля размещенного символа;
- как хранится угол поля;
- есть ли флаг auto-placement;
- есть ли признак ручного перемещения;
- есть ли различие между `Reference`, `Value` и пользовательскими полями;
- можно ли сохранить дополнительное состояние без изменения формата;
- как схема ведет себя при чтении старых файлов.

### Формат `.kicad_sym`

Выяснить:

- как библиотечные поля копируются в размещенный символ;
- какие свойства применяются при первом размещении;
- где и когда запускается авторазмещение;
- как хранится настройка auto-placement в библиотечном символе.

### Код поворота и зеркалирования

Найти:

- где выполняется transform символа;
- где трансформируются поля;
- есть ли отдельная ветка для полей с auto-placement;
- почему первое размещение работает иначе, чем последующие повороты;
- можно ли переиспользовать существующий алгоритм авторазмещения.

### Команда авторазмещения полей

Изучить:

- существующую команду авторазмещения;
- настройки, влияющие на результат;
- правила размещения `Reference` и `Value`;
- применимость алгоритма после rotate/mirror.

## Этап 2: классификация решений

После исследования разделить возможные правки на две группы.

### Без изменения формата

Подходит для текущего патча:

- при повороте/зеркалировании пересчитывать поля с включенным auto-placement;
- сохранять readable orientation для `Reference` и `Value`;
- изменить default auto-placement: `Reference` сверху, `Value` снизу;
- не сбрасывать auto-placement при перемещении всего символа;
- использовать существующие флаги и свойства без добавления новых данных в `.kicad_sch`.

### С изменением формата

Только для будущей версии KiCad:

- хранение manual overrides по ориентациям;
- отдельное состояние `Manual per orientation`;
- отдельное состояние `Manual locked/global`;
- хранение нескольких позиций/углов для одного поля.

Пример желаемых данных:

```text
field "Reference":
  0 degrees   -> position/angle
  90 degrees  -> position/angle
  180 degrees -> position/angle
  270 degrees -> position/angle
```

## Этап 3: MVP-патч без изменения формата

Если анализ подтвердит, что это возможно без изменения формата:

1. Для полей с включенным auto-placement:
   - пересчитывать положение после rotate/mirror символа;
   - держать текст читаемым;
   - не отключать auto-placement при перемещении всего символа.

2. Для `Reference` и `Value`:
   - `Reference` ставить сверху от bounding box символа;
   - `Value` ставить снизу от bounding box символа;
   - держать текст горизонтальным/readable.

3. Не менять формат `.kicad_sch`.

4. Не менять поведение полей, которые явно находятся в ручном/locked режиме, если такой режим уже можно определить по существующим данным.

## Этап 4: расширенная модель

После MVP и анализа формата принять решение:

- если orientation-local overrides можно реализовать без изменения формата, сделать вторым патчем;
- если нельзя, оформить proposal для разработчиков KiCad на следующую версию формата;
- явно отделить исправление readable auto-placement от расширения формата.

## Тестовые сценарии

Проверить минимум:

- резистор на 0, 90, 180, 270 градусов;
- конденсатор;
- диод;
- микросхема;
- разъем;
- power symbol;
- символы с несколькими units;
- скрытые поля;
- поля из database library;
- символы после ручного перемещения `Reference`;
- символы после ручного перемещения `Value`;
- возврат символа к исходной ориентации после нескольких поворотов;
- зеркалирование.

## Критерии успеха MVP

- Новый символ после размещения выглядит читаемо.
- После поворота символа `Reference` и `Value` не оказываются боком.
- `Reference` по умолчанию находится сверху, `Value` снизу.
- Перемещение всего символа не ломает режим авторазмещения.
- Старые схемы открываются без изменения формата.
- Поведение предсказуемо и не создает впечатление случайной перестановки подписей.
